# Keycloak + QQ-only Authentication 设计

## 背景

WotBTools 后续用户功能（leaderboard 记录归属、QQ 通知绑定、Blitz account 绑定、订阅、个人记录管理、风控）需认证系统。

采用 **Keycloak** 作为统一认证，WotBTools 自身不实现用户名密码登录、不保存密码。

## 核心决策

```
Keycloak + QQ Identity Provider only
```

- WotBTools 不做本地用户名/密码注册。
- Keycloak 负责认证。
- 用户只能通过 QQ IdP 登录。
- 后端只信任 Keycloak 签发的 JWT。
- PostgreSQL 保留本地 `app_user` 表，只存业务身份映射。

## 两个"用户"概念

| 概念 | 说明 |
|---|---|
| **Keycloak user** | 认证身份。不允许本地密码登录/注册/忘记密码，登录入口只保留 QQ IdP |
| **Wotbtools app_user** | 业务用户。关联 `leaderboard_record`、`notification_binding`、`blitz_account_binding`、设置、订阅。不存密码 |

## 登录流程

```
User clicks "Login with QQ"
  → frontend redirect to Keycloak
  → Keycloak redirect to QQ IdP
  → QQ authorize
  → Keycloak create/update federated user
  → frontend receive tokens
  → frontend call backend with Bearer token
  → backend validate Keycloak JWT
  → backend create/update app_user from JWT claims
```

## Keycloak Realm 设计

- Realm: `wotbtools`
- Client: `wotbtools-frontend` (public, Standard flow ON, Direct/Implicit OFF)
- Redirect URIs: `https://wotbtools.com/*`, `https://replay.wotbtools.com/*`, `http://localhost:5173/*`
- 关闭 User registration、Forgot password、Direct access grants
- Browser Flow 禁用 Username Password Form
- QQ 设为默认 IdP，目标：用户只看到"使用 QQ 登录"

## QQ IdP 接入风险

需验证：
1. QQ Connect 是否支持标准 OIDC
2. 是否可用 Keycloak Generic OIDC Provider
3. 是否可用 Generic OAuth2 Provider
4. 是否需自定义 Keycloak Identity Provider SPI

注意：不默认 QQ 是标准 OIDC；优先用 `openid`/`unionid`；不用用户手动输入的 QQ 号作主身份。

## 后端认证

```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://auth.wotbtools.com/realms/wotbtools
```

- 使用 Spring Security OAuth2 Resource Server（不用旧 Keycloak adapter）
- 请求头：`Authorization: Bearer <access_token>`

## app_user 表 (Flyway migration)

```sql
create table app_user (
    id bigint generated by default as identity primary key,
    keycloak_subject varchar(128) not null,
    qq_openid varchar(128),
    qq_unionid varchar(128),
    username varchar(100),
    display_name varchar(100),
    avatar_url text,
    created_at timestamp with time zone not null default now(),
    last_login_at timestamp with time zone,
    constraint uk_app_user_keycloak_subject unique (keycloak_subject),
    constraint uk_app_user_qq_openid unique (qq_openid)
);
```

## QQ 通知与 QQ 登录分离

- **QQ IdP 登录** = 识别用户
- **QQ Bot 通知** = 用户自愿接收消息

通知绑定流程：登录 → 点击绑定 → 生成绑定码 → 用户去 QQ Bot 发送绑定码 → Bot 回调后端 → 绑定到 app_user。

禁止：用户输入 QQ 号 → 系统直接给该 QQ 号发消息。

## notification_binding 表 (后续)

```sql
create table notification_binding (
    id bigint generated by default as identity primary key,
    user_id bigint not null references app_user(id),
    channel varchar(32) not null,          -- 'QQ'
    external_user_id varchar(128),
    bind_code varchar(16) not null,
    status varchar(32) not null,
    created_at timestamp with time zone not null default now(),
    confirmed_at timestamp with time zone,
    constraint uk_notification_binding_bind_code unique (bind_code)
);
```

## MVP 实施顺序

1. 部署 `auth.wotbtools.com` Keycloak
2. 创建 `wotbtools` realm + `wotbtools-frontend` client
3. 研究 QQ IdP 接入可行性
4. 禁用 Keycloak 本地登录/注册
5. 后端接 Spring Security Resource Server
6. 新增 `app_user` 表 (Flyway migration)
7. 新增 `GET /api/me`
8. 前端加 QQ 登录/退出/当前用户
9. 后续接 QQ Bot notification_binding

## 暂时不做

- 用户名密码/邮箱注册/忘记密码
- 自建登录表单
- 保存密码
- QQ Bot 发消息
- 强制 replay/leaderboard 登录
- admin UI
- 复杂 role/permission
- Google/GitHub/Discord IdP

## 验收标准

- Keycloak 运行在 `auth.wotbtools.com`
- 仅 QQ IdP 登录可用
- Keycloak 无用户名密码入口
- 后端验证 Keycloak JWT
- `/api/me` 需登录，可创建/更新 app_user
- 现有 replay/leaderboard 不受影响
- 数据库不存密码
- 可基于 app_user.id 扩展后续功能
