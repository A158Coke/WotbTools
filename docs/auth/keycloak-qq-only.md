# Keycloak + QQ-only Authentication 设计

## 背景

WotBTools 后续用户功能（leaderboard 记录归属、QQ 通知绑定、Blitz account 绑定、订阅、个人记录管理、风控）需认证系统。

采用 **Keycloak** 作为统一认证，WotBTools 自身不实现用户名密码登录、不保存密码。

## 核心决策

```
Keycloak + QQ Identity Provider only
```

- WotBTools 不做本地用户名/密码注册。
- Keycloak 负责认证。
- 用户只能通过 QQ IdP 登录。
- 后端只信任 Keycloak 签发的 JWT。
- PostgreSQL 保留本地 `app_user` 表，只存业务身份映射。

## 两个"用户"概念

| 概念 | 说明 |
|---|---|
| **Keycloak user** | 认证身份。不允许本地密码登录/注册/忘记密码，登录入口只保留 QQ IdP |
| **Wotbtools app_user** | 业务用户。关联 `leaderboard_record`、`notification_binding`、`blitz_account_binding`、设置、订阅。不存密码 |

## 登录流程

```
User clicks "Login with QQ"
  → frontend redirect to Keycloak
  → Keycloak redirect to QQ IdP
  → QQ authorize
  → Keycloak create/update federated user
  → frontend receive tokens
  → frontend call backend with Bearer token
  → backend validate Keycloak JWT
  → backend create/update app_user from JWT claims
```

## Keycloak Realm 设计

- Realm: `wotbtools`
- Client: `wotbtools-frontend` (public, Standard flow ON, Direct/Implicit OFF)
- Redirect URIs: `https://wotbtools.com/*`, `https://replay.wotbtools.com/*`, `http://localhost:5173/*`
- 关闭 User registration、Forgot password、Direct access grants
- Browser Flow 禁用 Username Password Form
- QQ 设为默认 IdP，目标：用户只看到"使用 QQ 登录"

## QQ IdP 接入风险

需验证：
1. QQ Connect 是否支持标准 OIDC
2. 是否可用 Keycloak Generic OIDC Provider
3. 是否可用 Generic OAuth2 Provider
4. 是否需自定义 Keycloak Identity Provider SPI

注意：不默认 QQ 是标准 OIDC；优先用 `openid`/`unionid`；不用用户手动输入的 QQ 号作主身份。

## 后端认证

```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://auth.wotbtools.com/realms/wotbtools
```

- 使用 Spring Security OAuth2 Resource Server（不用旧 Keycloak adapter）
- 请求头：`Authorization: Bearer <access_token>`

## app_user 表 (Flyway migration)

```sql
create table app_user (
    id bigint generated by default as identity primary key,
    keycloak_subject varchar(128) not null,

```

## QQ 通知与 QQ 登录分离

- **QQ IdP 登录** = 识别用户
- **QQ Bot 通知** = 用户自愿接收消息

通知绑定流程：登录 → 点击绑定 → 生成绑定码 → 用户去 QQ Bot 发送绑定码 → Bot 回调后端 → 绑定到 app_user。

禁止：用户输入 QQ 号 → 系统直接给该 QQ 号发消息。

## 战队管理（纯 Keycloak）

不建本地 clan 表。战队 = Keycloak group + role，JWT 携带。

### 模型

| 概念 | Keycloak 映射 | 命名规则 |
|---|---|---|
| 战队 (clan) | `Group` | `groupName = teamSlug`（如 `vortex`） |
| Leader | `Realm Role` | `{teamSlug}-leader` |
| Member | `Realm Role` | `{teamSlug}-member` |

### 权限流

```
管理员 (admin)
  → 调用 KC Admin API 创建 Group + Role
  → assign 某用户为 xx-leader（加入 Group + 赋予 leader role）

Leader
  → 邀请用户加入战队
  → KC: 用户加入 Group + 赋予 xx-member role

JWT claims 携带 group + role
  → 后端 @PreAuthorize("hasRole('vortex-leader')") 直接鉴权
```

### 不做什么

- 不建本地 `clan` / `clan_member` 表
- 不缓存战队成员列表（低频操作，调 KC API 即可）
- 战队名用 slug，避免中文/特殊字符

### 后端新增

- `KeycloakAdminService` — 封装 KC Admin REST API
- Admin endpoint：创建战队、assign leader（仅管理员）
- Leader endpoint：邀请成员、移除成员

## Blitz account 绑定

一个 Keycloak 用户可绑定一个 WoT Blitz 账号。不提供搜索/查找其他用户功能。

### 存储

- **KC 侧**：`user.attributes["blitz_account_id"]`（源，用户通过 KC Account API 自行设置）
- **DB 侧**：`app_user.blitz_account_id` 列（缓存 + unique 约束）

```sql
alter table app_user add column blitz_account_id bigint;
alter table app_user add constraint uk_app_user_blitz_account unique (blitz_account_id);
```

不建独立 binding 表。KC attribute 是源，DB 列仅用于唯一性校验和本地联查。

### 绑定流程

```
用户登录 → 前端 Profile 页显示输入框
  → 用户输入 account_id
  → 前端调 KC Account API: PUT /admin/realms/wotbtools/users/{id}
     (设置 attributes.blitz_account_id)
  → KC 更新 → 下次 token refresh 携带新 attribute
  → 后端 CurrentUserService 读 attribute → 校验唯一 →
     upsert app_user.blitz_account_id
  → 冲突返回 409 "该账号已被其他用户绑定"
```

### API

```
GET  /api/me    (返回含 blitzAccountId、teams、roles)
```

不提供 `POST/DELETE /api/me/blitz-account`——用户直接调 KC API。

### GET /api/me 响应

```json
{
  "id": 1,
  "displayName": "Coke",
  "avatarUrl": "https://...",
  "blitzAccountId": 123456789,
  "teams": ["vortex"],
  "roles": ["vortex-leader"]
}
```

## MVP 实施顺序

1. 部署 `auth.wotbtools.com` Keycloak
2. 创建 `wotbtools` realm + `wotbtools-frontend` client
3. 研究 QQ IdP 接入可行性
4. 禁用 Keycloak 本地登录/注册
5. 后端接 Spring Security Resource Server
6. 新增 `app_user` 表 (Flyway migration)
7. 新增 `GET /api/me`
8. 新增 `blitz_account_binding` 表 + 绑定 API
9. 前端加 QQ 登录/退出/当前用户 + Blitz 账号绑定 UI
10. 后续接 QQ Bot notification_binding

## 暂时不做

- 用户名密码/邮箱注册/忘记密码
- 自建登录表单
- 保存密码
- QQ Bot 发消息
- 强制 replay/leaderboard 登录
- admin UI
- 复杂 role/permission
- Google/GitHub/Discord IdP

## 验收标准

- Keycloak 运行在 `auth.wotbtools.com`
- 仅 QQ IdP 登录可用
- Keycloak 无用户名密码入口
- 后端验证 Keycloak JWT
- `/api/me` 需登录，可创建/更新 app_user
- 现有 replay/leaderboard 不受影响
- 数据库不存密码
- 可基于 app_user.id 扩展后续功能
