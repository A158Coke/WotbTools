# WotBTools User Profile 设计任务

## 1. 背景

WotBTools 当前使用 Keycloak 作为统一认证系统。

当前身份设计：

* 普通用户主要通过 QQ 登录。
* 管理员仍然保留 username/password 登录。
* Keycloak 负责认证、groups、realm roles。
* 军团 / 战队权限通过 Keycloak group + realm role 实现。
* 当前系统已有 leaderboard 数据，但还没有自己的 user/profile 表。
* 用户登录后，需要手动填写自己的 WoT Blitz account id。
* 当前无法通过官方 API 验证 account id 是否真实属于该用户。

因此，需要新增一个轻量的 `user_profile` 表，用于把 Keycloak 登录用户和 WotBTools 业务资料关联起来。

---

## 2. 核心设计原则

### Keycloak 负责

Keycloak 继续负责：

* QQ 登录
* 管理员 user/pass 登录
* 用户唯一身份 `sub`
* 用户 groups
* 用户 realm roles
* QQ federated identity

### WotBTools DB 负责

WotBTools 数据库负责：

* 用户站内 profile
* 用户填写的 WoT Blitz account id
* 用户填写的 WoT Blitz nickname
* 用户展示名 display name
* 和 leaderboard 数据的业务联动
* account id 唯一性约束

### 不把 account_id 只存 Keycloak attribute 的原因

一开始考虑过把 `wotb_account_id` 存在 Keycloak user attribute。

但最终不采用作为主数据源，原因是：

1. `wotb_account_id` 是 WotBTools 业务数据，不是认证数据。
2. 需要唯一性约束：同一个 WoT Blitz account id 不能被多个 WotBTools 用户使用。
3. Keycloak user attribute 不适合做强唯一约束。
4. 后续要和 leaderboard / league / boost / clan 业务数据联动，放数据库更自然。
5. Keycloak attribute 适合存小量身份属性，不适合承担业务数据唯一性和查询关系。

最终决定：

```text
Keycloak = 认证源 / 权限源
user_profile = WotBTools 业务用户资料源
```

---

## 3. 不需要绑定验证状态字段

不要添加：

```sql
wotb_account_bind_status VARCHAR(32) NOT NULL DEFAULT 'UNVERIFIED'
```

原因：

* 当前没有官方 API 支持。
* 无法可靠验证用户填写的 account id 是否真的属于该用户。
* 如果字段永远是 `UNVERIFIED`，它就是假状态。
* 不应该在模型里保留没有真实业务动作支撑的字段。

当前系统只能表达：

```text
用户声明自己的 WoT Blitz account id
```

系统不能表达：

```text
这个 account id 已经被认证属于该用户
```

因此，表中不要出现：

* `wotb_account_bind_status`
* `verified`
* `rejected`
* `verified_at`
* `verified_by`

前端文案也不要说“认证账号”或“已验证账号”。

推荐文案：

```text
设置游戏账号
填写游戏账号
关联游戏资料
```

不推荐文案：

```text
验证游戏账号
认证游戏账号
已认证账号
```

---

## 4. 表名

新增表：

```text
user_profile
```

不要使用表名：

```text
user
```

原因：

* `user` 在数据库和框架语境中容易混淆。
* `user_profile` 更准确表达“WotBTools 业务用户资料”。

## 5. 最终表结构

新增 Flyway migration：

```text
V5__create_user_profile.sql
```

SQL：

```sql
CREATE TABLE user_profile (
    id BIGSERIAL PRIMARY KEY,

    keycloak_user_id VARCHAR(64) NOT NULL UNIQUE,

    display_name VARCHAR(64),

    wotb_account_id BIGINT,
    wotb_nickname VARCHAR(64),
    wotb_server VARCHAR(32) NOT NULL DEFAULT 'CN',

    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT uk_user_profile_wotb_account
        UNIQUE (wotb_server, wotb_account_id),

    CONSTRAINT ck_user_profile_wotb_server
        CHECK (wotb_server IN ('CN'))
);
```

---

## 6. 字段说明

### id

```sql
id BIGSERIAL PRIMARY KEY
```

WotBTools 自己数据库中的 profile 主键。

后续业务表如果需要关联 WotBTools 用户，可以优先关联 `user_profile.id`。

---

### keycloak_user_id

```sql
keycloak_user_id VARCHAR(64) NOT NULL UNIQUE
```

对应 Keycloak JWT 里的 `sub`。

这是 Keycloak 登录用户和 WotBTools profile 之间的核心绑定字段。

示例 JWT：

```json
{
  "sub": "fe6c3d6c-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

对应：

```text
user_profile.keycloak_user_id = JWT sub
```

必须唯一。

一个 Keycloak user 只能对应一个 user_profile。

---

### display_name

```sql
display_name VARCHAR(64)
```

WotBTools 站内展示名。

它和 `wotb_nickname` 分开。

原因：

```text
display_name = WotBTools 站内展示名
wotb_nickname = 游戏内昵称
```

未来二者不一定相同。

初始值可以来自：

* Keycloak preferred_username
* Keycloak name
* QQ nickname
* 用户手动填写
* WoTB nickname

MVP 可以 lazy create 时用 token 中可用的名字作为默认值。

---

### wotb_account_id

```sql
wotb_account_id BIGINT
```

用户手动填写的 WoT Blitz account id。

允许为空。

原因：

* 用户刚 QQ 登录后可能还没有填写游戏账号。
* 未填写账号的用户也应该可以有 profile。

系统不验证 account id 归属，只保存用户声明值。

---

### wotb_nickname

```sql
wotb_nickname VARCHAR(64)
```

用户手动填写的 WoT Blitz 游戏昵称。

允许为空。

原因：

* 用户可能只知道 account id。
* 昵称可能变更。
* 当前没有官方 API 保证能自动获取或验证。

---

### wotb_server

```sql
wotb_server VARCHAR(32) NOT NULL DEFAULT 'CN'
```

当前只支持国服，所以默认并限制为：

```text
CN
```

用 CHECK constraint 限制：

```sql
CONSTRAINT ck_user_profile_wotb_server
    CHECK (wotb_server IN ('CN'))
```

未来如果支持其他服务器，可以扩展为：

```text
CN
ASIA
EU
NA
```

---

### created_at / updated_at

```sql
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
```

记录 profile 创建和更新时间。

注意：PostgreSQL 不会自动维护 `updated_at`。需要在应用层更新，或者未来添加 trigger。

MVP 建议应用层维护。

---

## 7. 唯一性约束

核心约束：

```sql
CONSTRAINT uk_user_profile_wotb_account
    UNIQUE (wotb_server, wotb_account_id)
```

含义：

```text
同一个服务器下，同一个 WoT Blitz account id 只能被一个 WotBTools 用户填写。
```

例如：

```text
CN + 123456789 只能属于一个 user_profile
```

注意：

* `wotb_account_id` 允许为 NULL。
* PostgreSQL 允许多个 NULL 出现在 UNIQUE 约束中。
* 所以未填写 account id 的用户不会互相冲突。

这正好符合需求：

```text
未绑定 account id 的用户可以有很多个。
填写了 account id 的用户必须唯一。
```

## 8. 不要在 user_profile 里存这些字段

当前不要添加：

```text
clan_tag
qq_openid
email
phone
role
group
wotb_account_bind_status
verified
verified_at
verified_by
```

原因：

### clan_tag

不要放进 `user_profile`。

军团身份已经由 Keycloak group 表达：

```text
/clans/{TAG}
```

如果在 DB 里再存 `clan_tag`，会产生两个真相来源：

```text
Keycloak group 说用户在 CHRD
DB clan_tag 说用户在 KSR
```

这会导致数据不一致。

### qq_openid

不要存。

QQ openid 已经在 Keycloak federated identity 里。

WotBTools 业务系统只需要知道：

```text
keycloak_user_id
```

### email / phone

不要存。

当前 QQ 登录不一定有可信 email。phone 更不应该碰。

### role / group

不要存。

权限和组织关系由 Keycloak 负责：

```text
realm_access.roles
groups
```

### verified 相关字段

不要存。

当前无法验证 account id 归属，所以不要设计验证状态。

## 9. 和 leaderboard 的联动

当前 leaderboard 表中已有：

```text
account_id
nickname
```

其中 `leaderboard_record.account_id` 是排行榜数据里的玩家账号 ID。

新增 `user_profile.wotb_account_id` 后，可以通过 account id 将登录用户和已有 leaderboard 数据关联起来。

关联链路：

```text
Keycloak JWT sub
  ↓
user_profile.keycloak_user_id
  ↓
user_profile.wotb_account_id
  ↓
leaderboard_record.account_id
  ↓
历史排行榜数据
```

也就是说：

```text
user_profile 是登录用户和已有 leaderboard 数据之间的桥。
```

---

## 10. leaderboard 联动方式

### 我的排行榜记录

```sql
SELECT lr.*
FROM leaderboard_record lr
JOIN user_profile up
  ON up.wotb_account_id = lr.account_id
WHERE up.keycloak_user_id = :keycloakUserId
ORDER BY lr.damage_dealt DESC;
```

### 我的最高伤害

```sql
SELECT lr.*
FROM leaderboard_record lr
JOIN user_profile up
  ON up.wotb_account_id = lr.account_id
WHERE up.keycloak_user_id = :keycloakUserId
ORDER BY lr.damage_dealt DESC
LIMIT 1;
```

### 我的各车辆最高伤害

```sql
SELECT DISTINCT ON (lr.tank_id)
    lr.*
FROM leaderboard_record lr
JOIN user_profile up
  ON up.wotb_account_id = lr.account_id
WHERE up.keycloak_user_id = :keycloakUserId
ORDER BY lr.tank_id, lr.damage_dealt DESC;
```

## 11. 不要给 leaderboard_record 添加外键

不要这样做：

```sql
ALTER TABLE leaderboard_record
ADD CONSTRAINT fk_leaderboard_user_profile
FOREIGN KEY (account_id) REFERENCES user_profile(wotb_account_id);
```

原因：

* leaderboard_record 是游戏账号成绩记录。
* 某个玩家的回放可能被上传，但该玩家还没有注册 WotBTools。
* leaderboard_record 可以先于 user_profile 存在。
* user_profile 只是声明某个登录用户关联某个 account id。
* 二者可以 join，但不应该强制外键。

正确关系是：

```text
leaderboard_record.account_id 是游戏账号 ID
user_profile.wotb_account_id 是用户声明的游戏账号 ID
二者软关联，不做 FK
```

---

## 12. 是否要给 leaderboard_record 加 server 字段

当前 MVP 不必须。

因为当前只支持国服 CN。

现在可以先用：

```sql
user_profile.wotb_account_id = leaderboard_record.account_id
```

未来如果排行榜支持多服务器，需要给 leaderboard_record 加：

```sql
server VARCHAR(32) NOT NULL DEFAULT 'CN'
```

然后关联变成：

```sql
user_profile.wotb_server = leaderboard_record.server
AND user_profile.wotb_account_id = leaderboard_record.account_id
```

并添加索引：

```sql
CREATE INDEX idx_leaderboard_record_server_account
    ON leaderboard_record (server, account_id);
```

当前阶段可以不做。

## 13. 登录后 profile 创建方式

使用 lazy create。

流程：

```text
用户 QQ 登录
  ↓
前端调用 GET /api/users/profile
  ↓
后端从 JWT 读取 sub
  ↓
查询 user_profile.keycloak_user_id = sub
  ↓
如果不存在，自动创建一条最小 profile
  ↓
返回 profile
```

初始 profile：

```text
keycloak_user_id = JWT sub
display_name = preferred_username 或 name
wotb_account_id = null
wotb_nickname = null
wotb_server = CN
```

这样不需要做 Keycloak event listener。

## 14. API 设计

统一使用 `/api/users/...` 风格。

不要混用：

```text
/api/me/...
/api/user/...
/api/users/profile/self
```

最终接口固定为：

```http
GET    /api/users/profile
PATCH  /api/users/profile
PATCH  /api/users/wotb-account
DELETE /api/users/wotb-account
```

这些接口全部表示“当前登录用户自己的资料”。

不要让前端传 `keycloakUserId`。

`keycloakUserId` 必须从 JWT `sub` 获取。

---

### 14.1 获取我的 profile

```http
GET /api/users/profile
```

认证：

```text
需要登录
```

后端逻辑：

```text
1. 从 JWT 读取 sub
2. 根据 sub 查 user_profile
3. 如果不存在，则 lazy create
4. 返回 profile
```

响应示例：

```json
{
  "id": 1,
  "keycloakUserId": "fe6c3d6c-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "displayName": "Coke",
  "wotbAccountId": 123456789,
  "wotbNickname": "CokeA158",
  "wotbServer": "CN"
}
```

---

### 14.2 更新我的基础 profile

```http
PATCH /api/users/profile
```

认证：

```text
需要登录
```

请求：

```json
{
  "displayName": "Coke"
}
```

第一版只允许用户更新：

```text
displayName
```

后端逻辑：

```text
1. 从 JWT 读取 sub
2. lazy create / find user_profile
3. 校验 displayName 长度
4. 更新 display_name
5. 更新 updated_at
6. 返回 profile
```

---

### 14.3 设置我的 WoTB account

```http
PATCH /api/users/wotb-account
```

认证：

```text
需要登录
```

请求：

```json
{
  "wotbAccountId": 123456789,
  "wotbNickname": "CokeA158",
  "wotbServer": "CN"
}
```

后端逻辑：

```text
1. 从 JWT 读取 sub
2. lazy create / find user_profile
3. 校验 wotbAccountId 是正数
4. 校验 wotbServer 当前只能是 CN
5. 检查 (wotb_server, wotb_account_id) 是否已被其他 profile 使用
6. 如果被占用，返回 409
7. 保存 wotb_account_id / wotb_nickname / wotb_server
8. 更新 updated_at
9. 返回 profile
```

---

### 14.4 解绑 WoTB account

```http
DELETE /api/users/wotb-account
```

认证：

```text
需要登录
```

后端逻辑：

```text
1. 从 JWT 读取 sub
2. find user_profile
3. 设置 wotb_account_id = null
4. 设置 wotb_nickname = null
5. 设置 wotb_server = CN
6. 更新 updated_at
7. 返回 profile
```

由于 `wotb_account_id = null`，唯一约束不会冲突。

## 17. JPA Entity 建议

```java
@Entity
@Table(
    name = "user_profile",
    uniqueConstraints = {
        @UniqueConstraint(
            name = "uk_user_profile_wotb_account",
            columnNames = {"wotb_server", "wotb_account_id"}
        )
    }
)
public class UserProfile {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "keycloak_user_id", nullable = false, unique = true, length = 64)
    private String keycloakUserId;

    @Column(name = "display_name", length = 64)
    private String displayName;

    @Column(name = "wotb_account_id")
    private Long wotbAccountId;

    @Column(name = "wotb_nickname", length = 64)
    private String wotbNickname;

    @Column(name = "wotb_server", nullable = false, length = 32)
    private String wotbServer = "CN";

    @Column(name = "created_at", nullable = false)
    private Instant createdAt;

    @Column(name = "updated_at", nullable = false)
    private Instant updatedAt;

    @PrePersist
    void onCreate() {
        Instant now = Instant.now();
        this.createdAt = now;
        this.updatedAt = now;

        if (this.wotbServer == null) {
            this.wotbServer = "CN";
        }
    }

    @PreUpdate
    void onUpdate() {
        this.updatedAt = Instant.now();
    }
}
```

```text

2. validate accountId > 0
3. validate server == CN
```

逻辑：

```text

2. set wotbAccountId = null
3. set wotbNickname = null
```

## 21. 安全规则

必须遵守：

```text
1. 用户只能修改自己的 profile。
2. keycloak_user_id 从 JWT sub 获取，不能从请求体传入。
3. 用户不能修改其他人的 profile。
4. wotb_account_id 必须是正数。
5. wotb_server 当前只能是 CN。
6. 同一个 wotb_server + wotb_account_id 只能被一个 profile 使用。
7. 不做 account id 归属验证。
8. 不存 QQ openid。
9. 不存 clan_tag。
10. 不把 user_profile 当权限源。
```

## 22. 和 Keycloak group / role 的边界

`user_profile` 不负责权限。

军团权限仍然来自 Keycloak：

```text
Group:
  /clans/{TAG}

Realm Roles:
  clan:{TAG}:member
  clan:{TAG}:leader
```

后端判断权限时，仍然读取 JWT：

```text
realm_access.roles
groups
```

`user_profile` 只负责 WotBTools 业务资料。

---

## 23. 和 leaderboard 的边界

`leaderboard_record` 不依赖 `user_profile`。

不要给 `leaderboard_record.account_id` 加外键。

正确方式是按需 join：

```sql
SELECT lr.*
FROM leaderboard_record lr
JOIN user_profile up
  ON up.wotb_account_id = lr.account_id
WHERE up.keycloak_user_id = :keycloakUserId;
```

这样可以支持：

```text
leaderboard 数据先存在
用户之后再注册并填写 account id
系统自动把他的历史 leaderboard 数据关联出来
```

## 25. 最终结论

最终模型：

```text
Keycloak:
  登录身份
  QQ federated identity
  groups
  realm roles

user_profile:
  keycloak_user_id
  display_name
  wotb_account_id
  wotb_nickname
  wotb_server
  created_at
  updated_at

leaderboard_record:
  account_id
  nickname
  battle / tank / damage data
```

联动方式：

```text
Keycloak JWT sub
  ↓
user_profile.keycloak_user_id
  ↓
user_profile.wotb_account_id
  ↓
leaderboard_record.account_id
  ↓
用户历史排行榜数据
```

一句话总结：

```text
Keycloak 负责登录身份和权限；user_profile 负责轻量业务资料；leaderboard_record 通过 account_id 与 profile 软关联。
```
